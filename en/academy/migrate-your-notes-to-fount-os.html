Bring your notes into Fount OS — with an AI you control.

An optional, hands-on route. One prompt turns an export from Notion, Obsidian, Evernote, Apple Notes and others into clean Fount OS markdown — files you own, on your own machine, with your originals untouched.

Guide · about 12 minutes · Works with Claude Cowork or any capable LLM

Fount OS has a built-in importer that does everything below for you — it's the easy path, and most people should start there. This guide is for when you want more control: your own folder shape, your own rules, and an LLM you can watch work. Same destination, hands on the wheel.

What "Fount OS native" actually means

There's no proprietary format to convert into. "Native" here just means plain Markdown files in a folder you own — the same files Fount OS reads directly, with nothing locked in. Four things make a note native:

Plain CommonMark. One .md file per note, UTF-8, ordinary Markdown in the body. No database, no export step later — the files are the data.

Light YAML frontmatter. An optional block at the top of a file, between two --- lines, holding a few fields like title, tags and dates. Fount keeps it deliberately simple: plain text, numbers, true/false, and lists — dates written as plain YYYY-MM-DD strings. Anything it doesn't recognise is preserved untouched, so you never lose metadata.

Wikilinks. Notes point at each other with [[Note title]] (or [[Note title|shown text]]) — the same syntax Obsidian uses.

Folders that mean something. Fount's typed views — the Lenses that show a folder as a list of people, a journal, or a set of decisions — key off where a note lives, not a hidden field. Put people in a people folder and the People lens lights up. Nothing moves; a Lens is a way of looking, not a container.

That's the whole target. The rest of this guide is just getting your export into that shape cleanly.

The one rule: work on a copy

Everything else is detail. This is the rule: the LLM reads from your export and writes to a new, separate, empty folder. It never edits your originals.

Give the model two folders — a read-only source and an empty destination — and make sure one isn't nested inside the other. Fount's own importer refuses to run when the source and destination overlap, precisely because pointing a migration at its own folder is how people accidentally duplicate everything. The prompt below builds in the same refusal.

Because you keep the originals, a migration is safe to throw away. If the first pass looks wrong, delete the destination folder and run it again with a tweaked prompt. Nothing is at stake.

The shape you're aiming for

Fount doesn't force a rigid structure — these are sensible defaults you can edit, mirroring how the app itself sorts an import. Adapt the folder names to your work; the only "special" one is the tasks file:

00-Inbox/                  anything you can't confidently classify (fallback)
10-People/                 one file per person or company
12-Meetings/               meeting notes
30-Projects/               one file (or subfolder) per project
50-Journal/Entries/        dated journal entries
60-Decisions/              one file per decision
80-System/Tasks/TASKS.md   a single running checklist of open tasks
99-Archive/                anything clearly old or inactive

If a group of notes doesn't fit, it keeps its own name as a top-level folder — don't force-fit, and don't invent extra buckets. When a note carries the right frontmatter, its Lens card gets richer. A person and a decision, for example:

---
type: person
name: Jane Okafor
company: Northwind
role: Head of Ops
email: jane@northwind.co
status: active
tags: [client, priority]
created: 2026-03-11
---

# Jane Okafor

First met at the Rotterdam meetup. Owns the Northwind rollout.
---
type: decision
title: Use plain markdown for all client notes
decided: 2026-05-02
status: settled
chosen: One .md file per note in a synced folder
ruled_out: [Notion, Airtable]
why: No lock-in; still readable in ten years
tags: [tooling]
---

# Use plain markdown for all client notes

We chose files over a database because the notes should outlive the app.

Every field is optional. A note with no frontmatter at all is still perfectly native — the extra keys just let the People, Decisions and Journal lenses render a fuller card. Leave out anything you don't have; never let the model invent values.

The base prompt (copy, then adapt)

This is the template. Paste it into Claude Cowork (pointed at your two folders) or any capable LLM, fill in the three {{...}} placeholders, and adjust the folder list to taste.

Base prompt · migrate to Fount OS 
You are helping me migrate my notes from {{SOURCE_APP}} into the "Fount OS" structure.
Fount OS is not a database — it's an ordinary folder of Markdown files that I own. Your
job is to convert my export into that shape faithfully and without ever touching my
originals.

## Inputs
- SOURCE (read-only, never modify): {{SOURCE_FOLDER}}
- DESTINATION (write only here — an empty, separate folder): {{DEST_FOLDER}}

## Rule 0 — never touch my originals
Treat SOURCE as read-only. Write only inside DESTINATION. If DESTINATION is inside
SOURCE, SOURCE is inside DESTINATION, or they are the same folder — stop and ask me to
choose a separate empty folder. Never move, rename, or delete anything in SOURCE.

## File format (match this exactly)
- One Markdown file per note. UTF-8, LF line endings, `.md` extension.
- The body is plain CommonMark. Don't invent HTML or custom syntax.
- Optional YAML frontmatter between two `---` lines at the very top. Keep it simple:
  - Only plain scalars (text, whole numbers, true/false) and lists of text.
  - No nested objects ( { } ), no block scalars ( | or > ).
  - Write dates as plain strings, YYYY-MM-DD (e.g. 2026-07-19) — never as typed YAML dates.
  - If a note already has frontmatter, keep every existing key in its original order,
    untouched. Only add keys — never reorder or drop keys you don't recognise.
- Links between notes use Obsidian-style wikilinks: [[Note title]] or [[Note title|shown text]].
  No [[Note#heading]], and no file paths inside [[ ]].

## Folders (sensible defaults — adapt them to me)
Place each note in the matching folder and keep filenames human-readable:

    00-Inbox/                  anything you can't confidently classify (fallback)
    10-People/                 one file per person or company
    12-Meetings/               meeting notes
    30-Projects/               one file (or subfolder) per project
    50-Journal/Entries/        dated journal entries
    60-Decisions/              one file per decision
    80-System/Tasks/TASKS.md   a single running checklist of open tasks
    99-Archive/                anything clearly old or inactive

If a source grouping doesn't map to one of these, keep it as its own top-level folder
using its existing (sanitised) name. Don't force-fit, and don't invent extra folders.

## Optional frontmatter per type (only when the info actually exists)
- People (10-People):   type: person · name · company · role · email · status · tags · created
- Decisions (60-Decisions): type: decision · title · decided (YYYY-MM-DD) · status · chosen · ruled_out (list) · why · tags
- Journal (50-Journal/Entries): type: journal · title · created (YYYY-MM-DD) · entry_type (daily | weekly | insight | quote | photo) · tags · mood
- Tasks: collect open to-dos as GitHub-style checkboxes in 80-System/Tasks/TASKS.md —
  "- [ ] task" / "- [x] done". Keep any [[links]] on the line.
Leave out any field you don't have. Never fabricate values.

## Links & collisions
1. First build an index of every note filename you will create. Only turn a reference
   into a [[Wikilink]] when its target exists in that index. If it doesn't resolve, keep
   the original text/link as-is and add it to an "unresolved links" list — never silently
   drop a link.
2. Never overwrite. If two notes would land on the same path, keep both: add -2, -3 … to
   the filename and record the rename.
3. If {{SOURCE_APP}} is Notion: strip Notion's 32-character hex id from every filename,
   folder name, and link target (e.g. "Roadmap 1a2b3c…7890.md" becomes "Roadmap.md"), and
   carry attachments across at their cleaned path so image links still resolve.

## How to run — propose first, then apply
1. Scan SOURCE and show me a PLAN: how many notes land in each folder, plus any name
   collisions or unresolved links you expect. Write nothing yet.
2. Wait for my "go".
3. On "go", create the files in DESTINATION.
4. Then write DESTINATION/MIGRATION-REPORT.md: notes written, count per folder, files
   skipped, every rename, and the full list of unresolved links.

Start with step 1.

What to change: the three {{...}} placeholders (your source app, and the two folder paths), and the folder list if your work doesn't look like people-projects-journal. Everything else is what keeps the output faithful — leave it in unless you know why you're removing it.

Run it in three passes

The prompt is written to propose before it writes — the same "preview, then confirm" beat the app uses. Don't skip it.

1 · Plan. The model scans your export and reports how many notes land where, plus the collisions and unresolved links it expects. Read this. If half your notes are heading for 00-Inbox, your folders need adjusting — fix the prompt, not the output.

2 · Spot-check. Ask it to fully convert just one folder first. Open a couple of the resulting .md files in any text editor. Frontmatter clean? Links look like [[wikilinks]]? Body intact? Good.

3 · Apply & verify. Give the "go", let it write the rest, then open MIGRATION-REPORT.md. It lists everything written, everything skipped, every renamed collision, and every link that couldn't be resolved. That last list is your punch-list — usually notes that referenced something you hadn't exported.

Notes for your app

Notion. Export as Markdown & CSV and hand the model the zip (or the unzipped folder). Two things matter: Notion appends a long hex id to every filename — the prompt strips it — and Notion databases come out as CSV, which the model should expand into one note per row, columns becoming frontmatter.

Obsidian. You may not need this at all: an Obsidian vault is already plain markdown with wikilinks, so you can point Fount OS straight at it and it reads in place. Only run a migration if you actually want to re-shape the folders — and if so, point the source at a copy of the vault.

Apple Notes. There's no clean markdown export, so notes arrive as rich text or HTML. Add a line to the prompt asking the model to flatten formatting to plain CommonMark. There's little metadata to preserve, so this is mostly about clean bodies and sensible titles.

Evernote. Export a notebook as .enex. Tell the model to split it into one note per entry and lift the title, created date and tags from the ENEX into frontmatter. Attachments should be written alongside their note and linked.

Capacities & Reflect. Both export markdown. Their objects and backlinks map naturally onto files and [[wikilinks]] — the base prompt handles them as-is; just name the app in the placeholder.

Open it in Fount OS

Point Fount OS at your new folder. It indexes in place — the files stay exactly where they are — and the Lenses read the folders you built: People becomes a directory, Journal a timeline, Decisions a log. If a folder doesn't light up the way you expected, it's a folder-placement thing, not a broken file; move the notes and re-open.

A closing note on privacy, since it's the point of doing this yourself. When you run the migration in a local or on-device model, your notes never leave your machine. Fount itself keeps its everyday work — search, indexing, Lenses — on-device by default, and only reaches for a cloud model when you explicitly ask. Choosing to migrate by hand, with a model you control, is the same principle applied one step earlier.
